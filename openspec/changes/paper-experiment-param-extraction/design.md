## Context

这是一个全新的全栈 Web 应用，目标是构建 AI 辅助的论文实验参数提取平台。目前科研团队完全依赖人工从论文中提取实验参数，效率低且无结构化存储。项目从零开始，无历史包袱，但需要在以下维度做到可用且健壮：文件解析、AI 集成、RBAC 权限、版本管理、并发控制。

## Goals / Non-Goals

**Goals:**
- 支持 PDF/DOCX/Markdown 三种格式的论文上传与自动解析
- 用户可自定义参数模板（YAML 格式），AI 基于模板提取结构化参数
- 完整的审核工作流：AI 提取 → 人工校验 → 入库
- 多版本参数记录，支持差异对比
- 参数库支持搜索、标签、导出（Excel/CSV）
- AI 自然语言问答查询参数库
- Dashboard 统计面板、通知中心、审计日志
- RBAC 三级权限（普通用户 / 审核员 / 管理员）
- 边界情况处理：模板引用保护、审核后归档、同名文件重命名、并发冲突提示、非法 JSON 重试

**Non-Goals:**
- 不实现 PDF 的 OCR 识别（依赖现有 PDF 文本提取库）
- 不支持实时协同编辑（仅提示冲突）
- 不实现论文全文搜索引擎（仅元数据+参数搜索）
- 不做多语言 i18n（初期仅中文）
- 不支持第三方 SSO 登录（初期用户名密码）

## Decisions

### 技术栈

| 层 | 选择 | 理由 |
|---|---|---|
| 前端 | React + TypeScript + Tailwind CSS + shadcn/ui | 组件丰富、类型安全、响应式 |
| 后端 | Node.js + Express + TypeScript | 与前端同语言、生态成熟 |
| 数据库 | PostgreSQL | 关系型数据、事务支持、JSON 字段 |
| ORM | Prisma | 类型安全、迁移管理方便 |
| 文件解析 | pdf-parse + mammoth + remark | 纯 JS 方案，无需额外服务 |
| AI 集成 | Anthropic API (Claude) | 结构化输出能力强 |
| 文件存储 | 本地文件系统（初期），可扩展 S3 | 简单起步 |

### 数据库核心表

```
papers          — 论文（文件名、路径、作者、DOI、状态、上传者）
templates       — 参数模板（名称、YAML 定义、创建者）
parameters      — 提取的参数（论文ID、模板ID、JSON 值、状态）
parameter_versions — 参数版本历史（参数ID、版本号、JSON 值、修改者）
users           — 用户（用户名、密码哈希、角色）
notifications   — 通知（用户ID、类型、消息、已读状态）
audit_logs      — 审计日志（用户ID、操作、目标类型、目标ID、变更内容）
```

### 状态机设计

论文状态流转：
```
待解析 → 解析中 → 解析成功 → 参数提取中 → 待审核 → 已入库
                 ↘ 解析失败（可重试）
```

参数状态流转：
```
AI已提取 → 审核中 → 审核通过（入库）→ 已归档
                  ↘ 审核拒绝 → 重新提取
```

### RBAC 权限模型

采用角色-权限码模式，中间件校验：

| 权限 | 普通用户 | 审核员 | 管理员 |
|---|---|---|---|
| 上传论文 | ✓ | ✓ | ✓ |
| 查看自己的论文 | ✓ | ✓ | ✓ |
| 查看所有论文 | | ✓ | ✓ |
| 管理模板 | | | ✓ |
| 审核参数 | | ✓ | ✓ |
| 导出数据 | ✓ | ✓ | ✓ |
| 管理用户 | | | ✓ |

### AI 工作流

1. 用户选择论文 + 模板，点击"开始提取"
2. 后端将论文文本 + 模板 YAML 发送给 LLM，要求返回 JSON
3. 校验返回结果是否为合法 JSON 且字段匹配模板
4. 非法 JSON 自动重试（最多3次），每次将错误信息回传给 LLM
5. 成功后参数进入"待审核"状态

### 并发冲突处理

- 参数审核采用乐观锁（version 字段）
- 更新时检查版本号，版本不匹配时拒绝并提示用户刷新

### 同名文件处理

上传时检测同名文件，自动追加序号：paper.pdf → paper(1).pdf → paper(2).pdf

## Risks / Trade-offs

- [AI 提取准确率不稳定] → 强制人工审核环节，AI 结果不可直接入库；非法 JSON 自动重试3次
- [大文件 PDF 解析耗时长] → 解析采用异步队列，前端轮询状态
- [LLM API 调用成本] → 模板约束缩小上下文范围，减少 token 消耗
- [并发审核冲突] → 乐观锁 + 版本冲突提示，而非悲观锁阻塞
- [单服务器文件存储不可扩展] → 初期使用本地存储，预留 S3 接口

## Open Questions

- 是否需要对论文全文做向量嵌入以支持语义搜索？（初期不做，后续可扩展）
- 参数值的单位标准化如何处理？（如 80°C vs 353K，初期原样保留）
