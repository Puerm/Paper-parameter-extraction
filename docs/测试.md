可以，而且我建议不要测试「Todo List」「博客系统」「聊天室」这种常规题目，因为现在绝大多数 Agent + LLM 都能做出来。

真正能测试 Vibecoding 工具能力的，是那种：

- 有业务规则
- 有多角色
- 有状态流转
- 有数据关联
- 有边界情况
- 有一定前端交互
- 需要 Agent 理解需求而不是 CRUD

下面这个题目比较适合。

# 题目：AI论文实验参数提取平台

## 项目背景

科研团队每天会阅读大量论文，需要从 PDF 中提取实验参数并形成结构化数据库。

目前完全依赖人工整理，效率低。

开发一个 AI 辅助的实验参数提取平台。

------

# 功能需求

## 1. 论文上传

支持上传：

- PDF
- DOCX
- Markdown

上传后自动解析。

状态：

- 待解析
- 解析中
- 解析成功
- 解析失败

------

## 2. 论文管理

列表字段：

| 字段       | 说明     |
| ---------- | -------- |
| 文件名     | 原始名称 |
| 上传时间   | 时间     |
| 作者       | 自动提取 |
| DOI        | 自动提取 |
| 状态       | 当前状态 |
| 提取参数数 | 数量     |

支持：

- 搜索
- 筛选
- 排序

------

## 3. 参数模板管理

用户可创建模板。

例如：

催化剂实验模板

```yaml
Catalyst:
Temperature:
Pressure:
Yield:
Reaction Time:
```

电池实验模板

```yaml
Electrolyte:
Capacity:
Cycle Life:
Voltage:
```

功能：

- 新建模板
- 编辑模板
- 删除模板
- 克隆模板

------

## 4. AI参数提取

用户选择：

- 论文
- 模板

点击：

```text
开始提取
```

系统调用 AI。

返回：

```json
{
  "Temperature":"80°C",
  "Catalyst":"Pd/C",
  "Yield":"92%"
}
```

------

## 5. 参数校验

AI结果不能直接入库。

需要人工审核。

页面：

左侧：

```text
原文内容
```

右侧：

```text
提取结果
```

支持：

- 修改
- 删除
- 新增

------

## 6. 多版本记录

每次修改都保存版本。

例如：

Version 1

```json
{
 "Yield":"92%"
}
```

Version 2

```json
{
 "Yield":"91%"
}
```

支持查看差异。

类似 Git Diff。

------

## 7. 参数库

最终确认的数据进入参数库。

支持：

- 搜索
- 标签
- 导出Excel
- 导出CSV

------

## 8. AI问答

用户输入：

```text
近三年所有催化剂实验中，
温度超过100℃且收率大于90%的有哪些？
```

系统自动查询参数库。

返回结果。

------

# 权限系统

## 普通用户

可以：

- 上传论文
- 查看自己的论文
- 提交审核

不能：

- 删除模板

------

## 审核员

可以：

- 审核提取结果
- 修改结果

------

## 管理员

拥有全部权限。

------

# 技术要求

## Dashboard

显示：

```text
论文总数
模板总数
参数总数
成功率
```

图表：

- 上传趋势
- 参数提取成功率
- 模板使用排名

------

## 通知中心

事件：

```text
解析完成
提取完成
审核通过
审核拒绝
```

支持未读计数。

------

## 审计日志

记录：

```text
谁
什么时间
修改了什么
```

------

# 隐藏测试点（检验Vibecoding能力）

### Case1

删除模板时：

如果已有论文引用该模板：

```text
不允许删除
```

------

### Case2

审核通过后：

参数被引用：

```text
禁止直接删除
```

只能归档。

------

### Case3

AI返回非法JSON

```text
{
 temperature:
}
```

系统自动重试3次。

------

### Case4

上传同名文件

系统自动重命名：

```text
paper.pdf
paper(1).pdf
paper(2).pdf
```

------

### Case5

两个审核员同时编辑

出现：

```text
版本冲突
```

提示用户刷新。

# 为什么这个题目适合测试 Vibecoding

它会同时考察：

- Agent任务拆解
- 数据库设计
- 状态机设计
- RBAC权限设计
- 前端页面生成
- API设计
- 异常处理
- 并发处理
- 版本管理
- AI工作流设计

如果你的工具能在不额外提示的情况下，把这些业务规则、状态流转、边界条件都实现出来，那基本已经超过市面上 80% 的 Vibecoding 产品了。
